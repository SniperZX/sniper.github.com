观察者模式

典型案例：订报纸

subject（出版社） 主题
observer（订阅者） 观察者

定义：

观察者模式定义了对象之间的一对多依赖，这样一来，当一个对象改变时，所有依赖它的对象都将收到通知并自动更新。

< !–more–>

优点：

主题和观察者是松耦合。

关于观察者的一切，主题只知道它实现了某个接口（observer接口），主题不需要知道观察者的具体类是谁，做了什么。
我们可以在任何时候增加观察者，主题唯一依赖的东西是一个实现observer接口的对象列表。

我们可以在运行时修改对象列表，主题不会有任何影响。

维护的是那个对象列表，注册时列表加入Observer，移除时列表删除Observer，
通知时，取出所有Observer，然后调用其update（）方法即可。

观察者模式的两种方式：

观察者分别有‘推’和‘拉’两种方式。

推，当主题发生改变时，主题主动的‘推’给各个观察者。
拉，主题提供getter方法以便观察者主动到主题中‘拉’取所需要的状态。

‘拉’的好处，可以控制通知的时机，有些不需要通知的变化，可以让观察者无感知。

JAVA API内置了观察者模式，并且提供了这两种方式。但是内置的Observable类和Observer违反了设计原则：针对接口编程，而不是针对实现编程。

策略模式

Posted on 2018-09-11 | In 设计模式 |

#策略模式

定义：

定义了算法族，分别封装起来，让他们之间可以相互替换。此模式可以让算法的变化独立于使用算法的客户

< !–more–>

个人理解：

一类事物存在着某些易变的行为 ，（比如打人用拳头还是刀，剑），根据设计原则一，把需要变化的地方提取出来，然后进行封装，然后把这种行为提取为一个接口，然后实现不同的行为，可以通过注入的方式，将行为动态的注入到那一类事物中，这样就可以实现行为发生变化，但不影响事物本身的实现。

例子：游戏中人物使用的武器就可以使用策略模式。提取出一个武器类，

这样人物变换武器时不会影响人物本身的变换，人物只用提供一个方法；

设计原则一：找出应用中需要变化之处，把它们独立出来，不要与那些不需要变化的代码混在一起

以上均为个人理解，如果错误，欢迎大家指正！！！
